# SER Token - Basic ERC20

## Опис проєкту

SER Token - це базовий ERC20 токен на блокчейні Ethereum, побудований з використанням OpenZeppelin ERC20. Це найпростіша версія токену з фіксованою кількістю, яка карбується одразу при деплої контракту.

Проєкт використовує **Hardhat** як середовище розробки для компіляції, тестування та деплою смарт-контрактів.

### Основний функціонал:
- Стандарт ERC20 (transfer, approve, transferFrom, balanceOf, totalSupply)
- Фіксована кількість токенів, створена при деплої
- Всі токени належать деплоєру
- Кастомна кількість десяткових знаків (12)
- Немає можливості карбувати нові токени або спалювати існуючі

### Властивості токену:
- **Назва**: SER Token
- **Символ**: SER
- **Десяткові знаки**: 12
- **Загальна кількість**: 1,400,000 SER (фіксована, створена при деплої)
- **Initial Supply**: Всі токени належать деплоєру

## Технічні деталі

- **Версія Solidity**: ^0.8.0
- **Ліцензія**: MIT
- **Залежності**: OpenZeppelin Contracts
  - ERC20

### Код контракту:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract SERToken is ERC20 {
    uint8 private _decimals;

    constructor() ERC20("SER Token", "SER") {
        _decimals = 12;
        _mint(msg.sender, 1400000000000000000000 * 10 ** uint256(_decimals));
    }

    function decimals() public view virtual override returns (uint8) {
        return _decimals;
    }
}
```

### Особливості реалізації:

#### constructor()
- Викликає конструктор ERC20 з назвою "SER Token" та символом "SER"
- Встановлює 12 десяткових знаків
- Карбує всю кількість токенів (1,400,000 SER) на адресу деплоєра
- Після деплою нові токени створити неможливо

#### decimals()
- Перевизначає стандартну функцію ERC20 (18 -> 12)
- Повертає 12 десяткових знаків

### Відмінності від розширеного SER Token:

| Функція | Basic Token | Advanced Token |
|---------|------------|----------------|
| Mint нових токенів | ❌ Ні | ✅ Так (owner) |
| Burn токенів | ❌ Ні | ✅ Так |
| Cap (обмеження) | ✅ Неявне (fixed supply) | ✅ Явне |
| Ownable | ❌ Ні | ✅ Так |
| Складність | 🟢 Проста | 🟡 Середня |

## Інструкція з компіляції та деплою в Remix IDE

### Крок 1: Відкрити Remix IDE
1. Перейдіть на https://remix.ethereum.org/
2. Дочекайтесь повного завантаження інтерфейсу

### Крок 2: Створити файл контракту
1. У лівій панелі оберіть "File Explorer"
2. Створіть файл `16.sol`
3. Скопіюйте код з файлу `contracts/16.sol` з цього репозиторію

**ВАЖЛИВО**: Remix автоматично завантажить OpenZeppelin контракти через import

### Крок 3: Компіляція контракту
1. Натисніть на іконку "Solidity Compiler"
2. Виберіть версію компілятора **0.8.0** або вище
3. Переконайтесь, що включено оптимізацію (200 runs)
4. Натисніть **"Compile 16.sol"**
5. Дочекайтесь завершення (може зайняти час через OpenZeppelin)
6. Перевірте зелену галочку успішної компіляції

### Крок 4: Деплой контракту
1. Натисніть на іконку "Deploy & Run Transactions"
2. У розділі **ENVIRONMENT** оберіть:
   - `Remix VM (Shanghai)` - для тестування
   - `Injected Provider - MetaMask` - для реальної мережі (Sepolia, Mainnet)
3. У розділі **ACCOUNT** оберіть акаунт, який отримає всі токени
4. У розділі **CONTRACT** оберіть `SERToken`
5. **VALUE** залиште `0` (конструктор не потребує ETH)
6. Натисніть **"Deploy"** (оранжева кнопка)
7. Підтвердіть транзакцію у MetaMask (для реальної мережі)

**ВАЖЛИВО**: Після деплою акаунт, з якого ви деплоїли, матиме ВСІ 1,400,000 токенів!

### Крок 5: Перевірка деплою
1. Контракт з'явиться у розділі **Deployed Contracts**
2. Розгорніть контракт
3. Натисніть **`totalSupply`** - має показати повну кількість токенів
4. Натисніть **`balanceOf`**, введіть адресу деплоєра - має показати всю кількість

## Послідовність тестування контракту

### ВАЖЛИВО: Підготовка
- Використовуйте мінімум 2 акаунти для тестів
- Перший акаунт (deployer) - має всі токени після деплою
- Другий акаунт - звичайний користувач

---

### Тест 1: Перевірка ініціалізації

**Мета**: Переконатися у правильній ініціалізації токену

1. Натисніть **`name`** (блакитна кнопка)
   - **Очікуваний результат**: `SER Token`

2. Натисніть **`symbol`**
   - **Очікуваний результат**: `SER`

3. Натисніть **`decimals`**
   - **Очікуваний результат**: `12`

4. Натисніть **`totalSupply`**
   - **Очікуваний результат**: Дуже велике число
   - У WEI: `1400000000000000000000000000000000`
   - Це = 1,400,000 SER з 12 десятковими знаками

---

### Тест 2: Перевірка балансу деплоєра

**Мета**: Підтвердити, що всі токени належать деплоєру

1. Розгорніть **`balanceOf`**
2. Введіть адресу першого акаунту (deployer)
   - Наприклад: `0x5B38Da6a701c568545dCfcB03FcB875f56beddC4`
3. Натисніть "call"
   - **Очікуваний результат**: `1400000000000000000000000000000000`
   - Це дорівнює totalSupply

---

### Тест 3: Перевірка балансу нового акаунту

**Мета**: Підтвердити, що інші акаунти не мають токенів

1. Розгорніть **`balanceOf`**
2. Введіть адресу другого акаунту
   - Наприклад: `0xAb8483Fb90B4c8183A35F8e4b2c5c0E9e8F9fE8E`
3. Натисніть "call"
   - **Очікуваний результат**: `0`

---

### Тест 4: Передача токенів (transfer)

**Мета**: Перевірити базову функцію передачі

**ВАЖЛИВО**: Переконайтесь, що вибрано перший акаунт (той, що має токени)

**Параметри для введення:**
1. Розгорніть функцію **`transfer`** (червона кнопка)
2. У поле `to` введіть адресу другого акаунту
   - Наприклад: `0xAb8483Fb90B4c8183A35F8e4b2c5c0E9e8F9fE8E`
3. У поле `amount` введіть: `100000000000000` (це 100 SER з 12 десятковими)
4. Натисніть **"transact"**

**Перевірка:**
1. Перевірте баланс першого акаунту через `balanceOf`
   - Має зменшитись на 100000000000000

2. Перевірте баланс другого акаунту через `balanceOf`
   - **Очікуваний результат**: `100000000000000` (100 SER)

3. Натисніть **`totalSupply`**
   - Має залишитись незмінним (токени не створюються і не зникають)

---

### Тест 5: Approve та TransferFrom

**Мета**: Перевірити механізм делегування

**Крок 1: Approve**
1. Вибрано **перший акаунт** (який має токени)
2. Розгорніть **`approve`**
3. У поле `spender` введіть адресу другого акаунту
4. У поле `amount` введіть: `50000000000000` (50 SER)
5. Натисніть **"transact"**

**Перевірка approve:**
1. Розгорніть **`allowance`**
2. У поле `owner` введіть адресу першого акаунту
3. У поле `spender` введіть адресу другого акаунту
4. Натисніть "call"
   - **Очікуваний результат**: `50000000000000`

**Крок 2: TransferFrom**
1. Переключіться на **другий акаунт**
2. Розгорніть **`transferFrom`**
3. У поле `from` введіть адресу першого акаунту
4. У поле `to` введіть адресу третього акаунту (або власну)
   - Наприклад: `0x4B20993Bc481177ec7E8f571ceCaE8A9e22C02db`
5. У поле `amount` введіть: `20000000000000` (20 SER)
6. Натисніть **"transact"**

**Перевірка:**
1. Баланс першого акаунту зменшився на 20 SER
2. Баланс третього акаунту збільшився на 20 SER
3. Allowance зменшився до `30000000000000` (50 - 20)

---

### Тест 6: Спроба передати більше ніж є на балансі

**Мета**: Перевірити захист від передачі більшої суми

1. Вибрано **другий акаунт** (який має 100 SER)
2. Розгорніть **`transfer`**
3. `to`: адреса третього акаунту
4. `amount`: `200000000000000` (200 SER - більше ніж є)
5. Натисніть **"transact"**

**Очікуваний результат:**
- Транзакція має **провалитись**
- Помилка: `ERC20: transfer amount exceeds balance`

---

### Тест 7: Перевірка Events

**Мета**: Переконатися, що події випромінюються

Після кожної транзакції перевіряйте консоль Remix:

1. **Transfer event** - після transfer() та transferFrom()
   - `from`: адреса відправника
   - `to`: адреса отримувача
   - `value`: кількість токенів

2. **Approval event** - після approve()
   - `owner`: власник токенів
   - `spender`: кому дозволено витрачати
   - `value`: кількість дозволених токенів

---

### Тест 8: Розподіл токенів між користувачами

**Мета**: Продемонструвати розподіл токенів

**Сценарій**: Перший акаунт розподіляє токени між кількома користувачами

1. З **першого акаунту** відправте:
   - 500 SER на другий акаунт
   - 300 SER на третій акаунт
   - 200 SER на четвертий акаунт

2. Перевірте баланси всіх акаунтів через `balanceOf`

3. Перевірте `totalSupply` - має залишатись незмінним

---

## Діаграма залежностей функцій

```
Деплой контракту
    ↓
Конструктор створює ВСІ токени
    ↓
Всі токени належать deployer
    ↓
Базові функції ERC20:
├── balanceOf(address) → перевірити баланс
├── totalSupply() → загальна кількість (незмінна)
├── transfer(to, amount) → пряма передача
├── approve(spender, amount) → дозвіл витрачати
├── allowance(owner, spender) → перевірити дозвіл
└── transferFrom(from, to, amount) → передача від імені
    ↓
ВАЖЛИВО: Нових токенів створити НЕ можна!
```

## Використання токену

### Сценарії використання:

1. **ICO/Token Sale**:
   - Deployer отримує всі токени
   - Розподіляє їх інвесторам через transfer()
   - Простий та прозорий механізм

2. **Airdrop**:
   - Deployer має всі токени
   - Масово розсилає токени користувачам

3. **Team Distribution**:
   - Розподіл токенів між членами команди
   - Фіксована кількість запобігає інфляції

4. **DeFi Integration**:
   - Додати токен у пул ліквідності
   - Використовувати для стейкінгу
   - Торгувати на DEX

## Переваги та недоліки

### Переваги:

1. ✅ **Простота** - мінімальний код, легко зрозуміти
2. ✅ **Безпека** - менше коду = менше вразливостей
3. ✅ **Прозорість** - фіксована кількість, немає прихованих функцій
4. ✅ **Низький gas** - деплой коштує менше
5. ✅ **Предсказуваність** - totalSupply ніколи не змінюється

### Недоліки:

1. ❌ **Негнучкість** - неможливо створити нові токени
2. ❌ **Немає burn** - неможливо зменшити supply
3. ❌ **Немає контролю** - будь-хто може передавати токени
4. ❌ **Втрачені токени** - якщо відправити на помилкову адресу, повернути неможливо

### Коли використовувати:

✅ **Використовуйте базовий токен, якщо:**
- Потрібен простий токен з фіксованою кількістю
- Плануєте ICO з однократним розподілом
- Немає потреби в додатковій функціональності
- Важлива мінімальна вартість деплою

❌ **НЕ використовуйте, якщо:**
- Потрібна можливість карбування нових токенів
- Потрібен механізм спалювання
- Потрібен контроль доступу (ownership)
- Плануєте складну tokenomics

## Можливі помилки та їх вирішення

### Помилка: "ERC20: transfer amount exceeds balance"
**Причина**: Спроба передати більше токенів ніж є на балансі
**Рішення**: Перевірте баланс через balanceOf перед передачею

### Помилка: "ERC20: insufficient allowance"
**Причина**: Спроба transferFrom без достатнього allowance
**Рішення**: Спочатку викличте approve з потрібною сумою

### Помилка: "ERC20: transfer to the zero address"
**Причина**: Спроба передати токени на нульову адресу (0x0)
**Рішення**: Перевірте адресу отримувача

### Помилка: "ERC20: approve to the zero address"
**Причина**: Спроба дати дозвіл нульовій адресі
**Рішення**: Введіть правильну адресу spender

### Помилка компіляції в Remix
**Причина**: OpenZeppelin контракти не завантажились
**Рішення**: Дочекайтесь завантаження (1-2 хвилини)

## Аналіз безпеки

### Позитивні моменти:

1. **OpenZeppelin ERC20** - аудований та перевірений код

3. **Необоротні транзакції** - якщо відправили помилково, не повернути

### Рекомендації для production:

1. Провести аудит безпеки

## Порівняння версій SER Token

У цьому репозиторії міститься **базова** версія SER Token. Є також **розширена** версія з додатковим функціоналом:

| Функція | Basic (цей репозиторій) | Advanced |
|---------|------------------------|----------|
| ERC20 стандарт | ✅ | ✅ |
| Fixed Supply | ✅ | ❌ |
| Mint | ❌ | ✅ |
| Burn | ❌ | ✅ |
| Cap | Неявний | ✅ Явний |
| Ownable | ❌ | ✅ |
| Decimals | 12 | 12 |
| Total Supply | 1,400,000 | 1,400,000 max |
| Складність | 🟢 Низька | 🟡 Середня |
| Gas Deploy | 🟢 ~800k | 🟡 ~1,200k |

**Вибір версії:**
- Якщо потрібна простота → Basic Token (цей репозиторій)
- Якщо потрібна гнучкість → Advanced Token
